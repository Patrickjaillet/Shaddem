# ShaderDemo — Feature & Capability Roadmap ("complete software improvement")

**Copyright (c) 2026 Patrick JAILLET**

Companion document to [`roadmap.md`](roadmap.md) (Python-port parity), [`roadmap-ux.md`](roadmap-ux.md) (interface architecture/performance/workflow), and [`roadmap-UI.md`](roadmap-UI.md) (visual craft/polish). Those three assume the software's *capabilities* are roughly fixed and improve how it's built, organized, and looks. This document is different: it's about **what the software can actually do** — closing real functional gaps and adding the capability that prompted it, compositing static images into a demo as first-class layers, the way demoscene "megademos" combine shader backgrounds, images/textures, text, and 3D objects into one sequenced piece rather than a single full-screen shader.

> Same living-document rule as the other three roadmaps: check boxes and log decisions in the same change that makes the change. Do not let this drift out of sync with the code.

**Baseline audit (2026-07-10, from the actual current `Rendering/`, `Timeline/`, and `Gui/LayersPanel.cs`)**:
- Image loading exists but is **single-slot**: `ShaderManager.LoadChannel0Texture` (`Rendering/ShaderManager.cs`) binds exactly one image as `iChannel0` for whatever shader is currently active — that's a shader *input*, not a compositable layer. `Gui/MediaPanel.cs` is the picker UI for it.
- The Layers system (`Gui/LayersPanel.cs`, `ShaderManager.Layers` : `List<Layer>`, `Rendering/Layer.cs`) is real and working, but every layer is a **shader pass**: `(ShaderName, BlendMode, Opacity, Enabled)`. There is no way to put a static image, text, or a 3D model into that stack — only another shader.
- `RenderPipeline.cs` already has the hard part built: `SceneFbo`/`LayerFbo`/`ComposeA`/`ComposeB` ping-pong framebuffers and a real `BlendLayer(baseFbo, layerFbo, BlendMode, opacity, target)` step using a `FragmentBlend` shader supporting Replace/Add/Multiply/Screen/Overlay/Normal/Difference/Exclusion. `ShaderManager.RenderFrame` loops `Layers`, renders each into `LayerFbo`, and blends it into the compose chain. This is the exact mechanism a new "image layer" or "text layer" needs to hook into — it does not need to be invented.
- `Timeline/ClipType.cs` already defines `Image`, `Text`, and `Model3D` as enum values, and `TimelineCanvas.cs` already color-codes them (`Theme.ClipImage`, `Theme.ClipText`, `Theme.ClipModel3D`) — but `Timeline/TimelineEngine.cs` only resolves `ClipType.Shader` and `ClipType.Effect` during playback. Adding those clips to a track today is legal but inert: nothing renders them. This is UI/data-model scaffolding with no execution behind it, not a working feature.
- `ModelRenderer` (3D) and `TextOverlay`/particles are separate, independent rendering paths inside `ShaderManager` today — they draw on top of the composed shader output, but are not part of the `Layers` blend stack, so they can't be reordered, blended, or opacity-controlled relative to a shader/image layer.

None of `roadmap.md`, `roadmap-ux.md`, or `roadmap-UI.md` cover multi-source layer compositing as a dedicated feature — this is genuinely new capability, not scope already claimed elsewhere.

## 0. Guiding principles (non-negotiable)

- [ ] Reuse the existing `RenderPipeline`/`BlendLayer` compositing mechanism — do not build a second, parallel compositor. An image layer and a shader layer should be able to sit in the same `Layers` list, reordered and blended the same way.
- [ ] A feature only counts as done when it's reachable from the real UI (Layers panel / Timeline) and driven end-to-end by a file the user actually picks via the native file dialog (`NativeFileDialog`, already used by `MediaPanel`) — not a hardcoded test-only path.
- [ ] Every item ships with a real build/run verification (the existing `--test-*` CLI harnesses, extended where needed, plus a real screenshot for anything visual) — matching `roadmap.md`/`roadmap-ux.md`/`roadmap-UI.md`'s own discipline.
- [ ] No feature ships that silently breaks an existing one — after every phase, re-run the full regression suite (`--test-docked-workspace`, `--test-effects-panel`, `--test-workflow-panels`, `--test-gui-panels`, `--test-full`, `--test-timeline-ui`, `--test-window-features`, `--test-particle-benchmark`) plus `dotnet test`.
- [ ] Prefer extending the existing `Layer`/`ClipType` data models over inventing new parallel concepts — a "megademo" is still just "several layers, sequenced over time," which is what `Layers` + `Timeline` already model for shaders; images/text/models need to join that model, not get their own.
- [ ] All new UI text stays English-only, matching `roadmap.md` §0's project-wide rule. No comments in source beyond the one copyright header line (`roadmap.md` §0), including in every new file this roadmap adds.

## 1. Image & media layers (the "megademo" feature)

### 1.1 Generalize `Layer` beyond shader-only
- [x] Extend `Rendering/Layer.cs` with a `SourceType` (`Shader`, `Image`, `Text`, `Model3D`) instead of assuming every layer is a shader name. `ShaderName` becomes one of several source fields (`ImagePath`, `TextContent`, `ModelPath`), populated depending on `SourceType`. (`Text`/`Model3D` are legal enum values but not yet rendered — same inert-until-wired pattern as `ClipType`, deferred to §1.5.)
- [x] Update `ShaderManager`'s layer loop (`RenderLayerChain`): for a `Shader` layer, behavior is unchanged (existing code path); for an `Image` layer, blit the loaded texture into `LayerFbo` via a new `RenderPipeline.RenderImageLayerPass` (textured full-screen quad pass with aspect-fit/position/rotation math, no shader program lookup), then fall into the same `RenderPipeline.BlendLayer` call already used today — same blend modes, same opacity slider, no new compositing logic.
- [x] Reuse `TextureLoader` (already used by `MediaPanel`'s single-slot `iChannel0` loading, via the same `ShaderManager.LoadChannel0Texture` pattern) for image-layer textures, cached per path in `ShaderManager._layerImageCache` — one texture-loading code path, not two.

### 1.2 Image layer placement & framing controls
- [x] Per-image-layer transform: position (X/Y offset), scale, and rotation, plus a fit mode (Stretch / Fit / Fill / Center) — stored on `Layer` (`PositionX/Y`, `Scale`, `Rotation`, `FitMode`), edited via `LayersPanel`'s image-layer editor.
- [x] Respect existing blend modes and opacity from the shader-layer system unchanged — an image layer sits in the same `Layers` list and goes through the same `BlendLayer` call, reorderable/blendable identically to a shader layer.

### 1.3 `Gui/LayersPanel.cs`: "Add Image Layer"
- [x] Add an "Add Image Layer" action alongside the existing "Add Layer" (shader) action, opening the same native file dialog already wired up in `MediaPanel` (`NativeFileDialog.OpenFile`/`BuildFilter` with `MediaPanel.ImageExtensions`, promoted to `internal` and reused directly — no duplicate filter array, no new dialog code).
- [x] Show a small thumbnail per image layer in the layer list (`MediaPanel.GetThumbnail`/`_thumbnailCache`/`_asyncLoader` promoted to `internal` and called directly from `LayersPanel`, so there's still exactly one thumbnail cache/async-decode pipeline in the app, not two).

### 1.4 Wire `ClipType.Image` into the Timeline for real
- [x] `Timeline/TimelineEngine.cs`: added `ApplyImageLayers(ShaderManager, currentTime)` — for each active `Image` clip (via the existing `GetFilteredActiveClips`), it upserts a corresponding `Layer` (`Layer.CreateImage`, tagged `IsTimelineManaged = true`) into `ShaderManager.Layers`, tracked by clip identity in an internal `Dictionary<TimelineClip, Layer>` so it never touches layers the user added manually via `LayersPanel`; when a clip goes inactive its layer is removed. Wired into the main loop in `Program.cs` alongside the existing `ApplyEffects`/`ApplyShader` calls. This turns the previously-inert `ClipType.Image` enum value into a real feature: sequencing images in and out over time.
- [x] Fade in/out at clip boundaries: **course-correction from the original wording** — research during implementation found `TimelineEngine` has no crossfade/transition logic of its own to reuse; the only existing crossfade (`ShaderManager.StartTransition`/`IsTransitioning`) is a single-active-shader hard-cut-vs-crossfade state machine triggered manually from `EffectsPanel`, not something `TimelineEngine.ApplyShader` invokes automatically, and it doesn't fit a multi-layer stack. Instead, `ApplyImageLayers` computes opacity as `baseOpacity * min(fadeInFactor, fadeOutFactor)` using the clip's own `Start`/`End` and configurable `fade_in`/`fade_out`/`opacity` clip params (default 0.5s fades) — reusing the same `TimelineClip.TryGetDouble`/`Params`-based configuration convention `ApplyEffects` already uses for effect clips, rather than a second parallel config mechanism. Verified numerically via `--test-timeline-image`: opacity measured at 0.16 during fade-in, 0.8 steady mid-clip, 0.32 during fade-out, for a clip with `opacity=0.8, fade_in=0.5, fade_out=0.5`.

### 1.5 Text and 3D-model layers join the same stack (stretch, but same mechanism)
- [x] **Go decision made** (2026-07-10) after sizing both halves honestly: Text was a small extension (`TextOverlay.Render` already accepted a target `Framebuffer`); Model3D was meaningfully bigger (no per-layer data model existed — `Model3DState` was a single global instance). Shipped both, full scope, per explicit product decision.
- [x] **Text**: `ShaderManager` now manages a single internal, `IsTimelineManaged` `Layer` (`SourceType.Text`) that appears/disappears in `Layers` exactly when a `Text` clip is active (`UpdateTextLayer`, called from `RenderLayerChain`, using the same `Timeline.GetFilteredActiveClips(...).LastOrDefault(c => c.Type == ClipType.Text)` lookup the old code used) — no `LayersPanel` "Add Text Layer" button, since text has always been authored via Timeline clips only, not manual layer picking. `TextOverlay.Render` now targets a cleared `Pipeline.LayerFbo` instead of `composed` directly, then goes through the same `BlendLayer` call as every other layer type. The old post-bloom/post-accumulation `TextOverlay.Render` call in `RenderFrame` was deleted. **Accepted, user-approved behavior change**: text is no longer unconditionally "always crisp, always on top" — it's now reorderable/blendable like any other layer and *is* subject to bloom/accumulation if placed before them in the stack (previously it ran after unconditionally). Caption content/style stay entirely clip-driven (`LayersPanel` shows a read-only notice pointing to the Timeline clip for a Text layer, matching the pattern already used for timeline-managed Image layers).
- [x] **Model3D**: `Layer` gained a `Model3DState ModelState` field (reusing the existing class instead of duplicating its 8 fields) and a `Layer.CreateModel3D(modelPath, texturePath, blendMode, opacity)` factory. Since `ModelRenderer` only ever holds one model/texture GPU resource slot (confirmed by reading `ModelRenderer.cs`), multiple simultaneous Model3D layers each get their own cached `ModelRenderer` instance (`ShaderManager._layerModelRenderers`, keyed by `Layer` reference, reloading only when the layer's model/texture path changes) rather than fighting over one shared slot. `RenderLayerChain` renders each into a cleared `LayerFbo` then blends normally. `LayersPanel` gained "Add Model Layer" (reusing `Model3DPanel.ObjExtensions`/the existing texture-picker filter, promoted to `internal`) and a full per-layer editor (position/rotation/scale/auto-rotate/wireframe, mirroring `Model3DPanel`'s existing global-model controls). `LayerPersistence` extended to (de)serialize `ModelState`. The pre-existing single global `ShaderManager.Model`/`ModelState`/`Model3DPanel` "quick overlay" feature was left untouched — this is a genuinely separate, additive mechanism, not a replacement, so no regression risk to the existing single-model workflow. **Not done** (confirmed out of scope by §1.5's own wording, which only asks for the `LayerFbo` rendering generalization, not clip-driven sequencing): no `TimelineEngine.ApplyModelLayers`/`ClipType.Model3D` timeline wiring — Model3D layers are manual/static only, same tier as manually-added Image layers before §1.4 wired those to the Timeline. Flagged as a legitimate future §1.4-for-Model3D extension, not silently assumed done.
- [x] **Bug caught during verification**: the auto-created internal Text layer defaulted to `BlendMode.Replace` (the `Layer` constructor's default), which silently blanked the entire frame to black except the caption glyphs — caught via the new `--test-text-model-layer` screenshot (first attempt rendered solid black), fixed by explicitly passing `BlendMode.Normal` when creating the internal text layer. Left as a reminder that every internally-constructed `Layer` needs an explicit blend mode, not the constructor default.
- [x] Verified via `dotnet test` (26/26), the full CLI regression suite (`--test-full`, `--test-timeline-ui`, `--test-window-features`, `--test-effects-panel`, `--test-docked-workspace`, `--test-workflow-panels`, `--test-gui-panels`, `--test-font-glyphs`, all pass / no new exceptions), re-running the §1.1–1.6 harnesses (`--test-image-layer`, `--test-timeline-image`, `--test-template-layer`, still pass — no regression from the Text/Model3D changes), and a new `--test-text-model-layer <modelPath>` harness asserting exactly one Text layer and one Model3D layer exist while both are active, that the Text layer is correctly removed once its clip's time window ends, and a real screenshot showing a shader background + rotating textured cube + "Megademo" caption all composited together in one stack. **Unrelated flake caught along the way**: `shaders/Alien_Sun.glsl` (used by §1.6's "Image Layer Showcase" template) disappeared from the source directory mid-session — not caused by this work (confirmed against the exact list of 44 files removed in the earlier shader-compatibility cleanup, which did not include it); swapped the template to `Cosmic_Space.glsl`, a currently-verified-present shader, and re-confirmed `--test-workflow-panels` is back to the same 12 pre-existing unrelated template failures as before.

### 1.6 Template & export coverage
- [x] At least one new built-in `DemoTemplate` (`Settings/TemplatesService.cs`) that actually demonstrates image-layer compositing: "Image Layer Showcase" (`Alien_Sun.glsl` background + `Layer.CreateImage(...)` blended `Screen` at 0.5 opacity, `Fit`-scaled to 0.6). `DemoTemplate` gained a `Func<IEnumerable<Layer>>? Layers` field; `TemplatesService.Apply` now does `manager.Layers.RemoveAll(l => !l.IsTimelineManaged); manager.Layers.AddRange(template.Layers())` so applying a template resets the layer stack to the template's preset without touching timeline-owned layers (§1.4's ownership tag made this safe). No `TemplatesPanel` changes needed — it already iterates `TemplatesService.BuiltIn` generically. The referenced image (`assets/screenshot-workspace.png`) is now copied to the build output (`assets\*.png` added to `ShaderDemo.App.csproj`, same pattern as `shaders\*.glsl`/`models\*.obj`) so the template resolves at runtime via `AppContext.BaseDirectory`.
- [x] Confirmed video/frame export captures the fully composited output including image layers — verified for real, not assumed. Code-level proof: in `ShaderManager.RenderFrame` (`ShaderManager.cs:252,256`), `LastComposedFrame = composed;` and `Recorder.WriteFrame(composed.ReadPixelsRgb());` both read the identical `composed` object from the same call, after `RenderLayerChain` and all post-effects/overlays — there is no code path where video export and the screenshot/preview path could see different framebuffers. Empirical proof: three separate harnesses (`--test-image-layer`, `--test-timeline-image`, `--test-template-layer`) all screenshot `LastComposedFrame` and visibly show the image layer composited in. **Honest limitation**: ffmpeg is not installed in this environment, so the actual `.mp4`/`.gif` byte output could not be probed end-to-end here (`--test-video-export` exists for that but requires a real ffmpeg binary) — the video path was verified by code identity with the already-proven screenshot path, not by decoding an exported video frame. If ffmpeg becomes available, re-run `--test-video-export <ffmpegPath>` with an image layer added to close that last gap.

## 2. Timeline & automation gaps

- [x] Per-layer keyframing: added `ClipType.LayerAutomation` — a clip whose `Resource` names a target `Layer.Name` (a new stable, user-editable label field, defaulting to `Layer1`/`Layer2`/…) and whose `Params` carry `property` (`opacity` or `enabled`) plus `value`/`start_value`/`end_value`/`easing`, resolved by `TimelineEngine.ApplyLayerAutomation` reusing the *exact* existing `TryGetInterpolatedScalar`/`NormalizedProgress`/`ApplyEasing` helpers `ApplyEffects` already uses — same mechanism, not a parallel one, per `roadmap.md` §5's own note that there's no dedicated automation-clip type distinct from `TimelineClip` (an `Effect` clip *is* the automation system; `LayerAutomation` mirrors that shape for `Layer` targets). Only targets non-`IsTimelineManaged` layers (manually added via `LayersPanel`), so it can't fight with `ApplyImageLayers`/`ApplyModelLayers`'s own opacity writes on their own timeline-owned layers. `LayersPanel` gained an editable "Layer Name" field (for non-timeline-managed layers) so a layer can be referenced from a clip; `TimelinePanel`'s "Add clip" UI gained a target-layer combo + property/start/end fields when `LayerAutomation` is selected. Blend-mode animation was explicitly scoped out — the roadmap item's problem statement mentioned it but the concrete ask only listed opacity/enabled, and a discrete enum isn't naturally keyframe-interpolated anyway.
- [x] Audit `Timeline/TimelineEngine.cs`: found two more genuinely-inert `ClipType`s beyond the already-fixed `Image` — **`Model3D`** (legally placeable on the "Video" track, never resolved during playback; `Model3DState` was a single global instance with no clip-driven equivalent) and **`Music`** (legally placeable on the "Audio" track; actual audio playback was entirely separate — a manual file-picker + Play button in `AudioPanel` totally ignoring any `Music` clip's start time or duration). Both fixed, not just documented: `TimelineEngine.ApplyModelLayers` mirrors §1.4's `ApplyImageLayers` pattern exactly (create/track/remove a `Layer.CreateModel3D` per active `Model3D` clip, with `Params`-driven position/rotation/scale/auto-rotate/texture/blend/fade, reusing the same `_imageClipLayers`-style dictionary pattern); `TimelineEngine.ApplyMusicClip` starts playback via `ShaderManager.Player` (a new `AudioPlayer.Play(path, volume, startOffset)` overload seeks to `currentTime - clip.Start` via NAudio's `AudioFileReader.CurrentTime`, so starting a clip mid-timeline seeks to the right position instead of always restarting from 0) when a `Music` clip becomes active, and stops it when none is. **Honest gap**: no continuous scrub-resync — if the playhead jumps while the *same* clip stays active (e.g. dragging the timeline cursor), audio position isn't re-synced, only re-seeked on clip transition; `AudioPlayer` has no position-read API to detect drift cheaply. Both fixes wired into `Program.cs`'s main loop alongside the existing `Apply*` calls, and into `TimelinePanel`'s "Add clip" UI (target-layer/model/audio-file fields per type).

## 3. Rendering & performance

- [x] Multi-image-layer performance: added `--test-layer-benchmark <imageDir>`, mirroring `--test-particle-benchmark`'s structure (30 samples/count after warmup) but scaling simultaneous image layers 1→2→4→8→16→32, cycling across distinct source images (not one repeated file, so the layer cache can't mask real per-texture cost) and reporting both CPU submission time and **real GPU time** via `ShaderManager.Profiler`'s existing GL timer-query "Scene" section (which wraps `RenderLayerChain`, i.e. the actual `LayerFbo` ping-pong/blend cost) plus `GpuResourceTracker.EstimatedVramBytes`. Measured result: GPU "Scene" cost plateaus at ~9ms from 2→32 layers in this environment — the base shader's own raymarching cost dominates, and the per-layer `RenderImageLayerPass`+`BlendLayer` passes (simple textured-quad + blend shaders) add negligible marginal GPU cost even at 32 simultaneous layers; VRAM scales correctly with distinct-texture count (20.5→90.2 MB across 9 unique source images, correctly plateauing once layers start repeating already-cached textures). **Measurement caveat, not hidden**: the very first ("1 layer") GPU timing reading is a consistent ~138ms outlier — a cold-start artifact from the first-ever GL timer query after context creation, not a real per-layer cost; excluded from the "how many layers before frame time suffers" conclusion above and called out explicitly rather than silently dropped from the reported numbers.
- [x] Texture memory: `TextureLoader.Load` gained an optional `maxDimension` parameter and a new public `DownscaleIfNeeded` helper (simple box/nearest-neighbor resize on the decoded RGB byte buffer, no new package dependency) that clamps a source image's longest edge before it ever reaches `Texture.Create`/VRAM. `ShaderManager` gained a `MaxLayerImageDimension` property (default 2048px) applied in `GetOrLoadLayerImage`, and `LayersPanel` exposes it as a live slider (512–4096px) with an estimated-VRAM readout, so it's a real user-facing option per the roadmap wording, not a silent internal constant. Verified two ways: 5 new unit tests (`TextureLoaderTests.cs`, 26→31 total) on the pure downscale math (aspect ratio preserved, byte-length correctness, portrait/landscape, disabled-clamp passthrough via reference equality) and a new `--test-texture-budget <largeImagePath>` end-to-end harness that loads a real generated 4000×3000 (34.3 MB raw) PNG through the actual `ShaderManager`/GL path and confirms the clamp reduces it to 1024×768 (2.2 MB raw, ~15.6x reduction) — not just unit-tested math, the real load path. `AsyncTextureLoader` (the thumbnail-decode path) was deliberately left unclamped since thumbnails are already small/decoded off the main texture-VRAM budget the roadmap item is about.

## 4. Export & output quality

- [x] Verified still-image export reflects the full layer stack including image/text/model layers: every `ScreenshotService.Save(...)` call site in the app (`ExportPanel.cs`, `SystemPanel.cs`, `CommandPalette.cs`, `Program.cs`, and every CLI test harness) uses `manager.LastComposedFrame ?? manager.Pipeline.SceneFbo` — the same post-`RenderLayerChain` buffer proven in §1.5/§1.6/§2's screenshots to already contain Image, Text, and Model3D layers composited together (see `--test-text-model-layer`'s screenshot showing shader + rotating model + caption all in one still capture). No code change needed; this was already true by construction once §1.5 shipped.
- [x] Checked the export-resolution-differs-from-preview premise directly and found it doesn't apply as stated: **there is no export-at-custom-resolution feature in this codebase today** — `ExportPanel`/`QuickExportPanel`/`DemoWizardPanel` all explicitly hardcode `manager.Pipeline.Width/Height` for export and label it "(current window)"/"(matches the current window)" in the UI; `VideoRecorder.Start`/`ScreenshotService.Save` never call `Pipeline.Resize`. So today export literally cannot differ in resolution from the live preview — the risk this checklist item worried about doesn't exist yet as a shipped path. Rather than mark it "N/A" without evidence, verified the underlying building block a future export-resolution feature would depend on: `RenderPipeline.RenderImageLayerPass`'s fit-mode math reads `target.Width`/`target.Height` live off the `Framebuffer` (not a cached/baked constant), and `RenderPipeline.Resize`/`RecreateInternalBuffers` correctly rebuilds `LayerFbo` (and `SceneFbo`/`ComposeA/B`) at new dimensions. Proved this empirically with a new `--test-export-resolution <imagePath>` harness: renders a `Fit`-mode image layer at 640×360, screenshots, calls `ShaderManager.Resize(1920, 1080)`, asserts `Pipeline`/`LayerFbo`/`LastComposedFrame` all report the new dimensions, renders and screenshots again — both screenshots show correctly proportioned framing with no exceptions. **Honest scoping note**: this verifies the resize mechanism is resize-safe, not a shipped "export at a different resolution than preview" feature — that would still need real UI work (width/height fields in `ExportPanel`, a resize-before-record/restore-after choreography, and interaction with `RenderScale`/adaptive resolution) which is out of scope for a verification item and isn't claimed done here.

## 6. Suggested phasing

1. **§1.1–§1.3** — generalize `Layer`, wire image-layer rendering through the existing `BlendLayer` mechanism, add the "Add Image Layer" UI. This alone delivers the headline ask: images composited into a demo like a megademo layer.
2. **§1.4** — wire `ClipType.Image` into the Timeline so images can be sequenced in/out over time, not just statically stacked.
3. **§1.6** — a template proving it end-to-end, plus export verification, so the feature is discoverable and confirmed to survive rendering/export.
4. **§2–§4** — automation, performance, and export-quality follow-ups once the core feature is stable and real usage patterns (how many layers, how large the images) are known.
5. **§1.5** — text/3D-model layer generalization, explicitly deferred until the image-layer path (the simpler of the three) has shipped and been verified, so the harder two-subsystem integration isn't attempted before the pattern is proven once.
6. **§5** — close the loop, same as every other roadmap in this project.
