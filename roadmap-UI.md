## 0. Guiding principles (non-negotiable)

- [ ] Every visual decision has a *reason* tied to hierarchy, focus, or feedback ŌĆö never decoration for its own sake. If a gradient/shadow/animation doesn't help the user understand state or where to look, cut it.
- [ ] Consistency over novelty: one spacing grid, one radius scale, one elevation system, one motion curve, applied everywhere ŌĆö not 33 panels each reinventing their own polish.
- [ ] Never sacrifice the Simple/Advanced/docked-workspace architecture already shipped (`roadmap-ux.md` ┬¦1.2/┬¦1.3) to chase visual polish ŌĆö this roadmap re-skins and refines what exists, it does not re-architect.
- [ ] Every item ships with a real build/run verification (screenshot or harness), matching `roadmap.md`/`roadmap-ux.md`'s own discipline ŌĆö no "looks right in my head" checkmarks.
- [ ] Respect the existing licensing constraints already hit and logged (`roadmap-ux.md` ┬¦1.1: no bundled icon font, `System.Drawing.Common` for text, Segoe UI/Consolas referenced from `C:\Windows\Fonts\` not bundled) ŌĆö new assets (icon font, splash art, logo) must be either self-authored, public-domain, or under a license compatible with an unsigned freeware distribution; document the source/license of anything added.
- [ ] All copy/UI text stays English-only, matching `roadmap.md` ┬¦0's project-wide rule.

### 1.2 Elevation & depth system
- [x] New `Gui/Elevation.cs`: `DrawShadow(pos, size, rounding, layers)` draws a soft layered shadow (5 nested semi-transparent rounded rects, offset down-right, decreasing alpha outward) onto `ImGui.GetBackgroundDrawList()` ŌĆö chosen over draw-list channel-splitting (the more "correct" but fragile technique for true per-window z-ordered shadows) since this project's floating windows are effectively mutually exclusive in practice (rarely two open at once), making the simpler background-draw-list approach a reasonable, low-risk approximation. Queries `ImGui.GetWindowPos()/GetWindowSize()` *after* `Begin()` each frame (correct even for `ImGuiWindowFlags.AlwaysAutoResize` windows like `QuickExportPanel`, since ImGui composites draw lists by list identity, not call order within the frame ŌĆö background always renders behind regardless of when in the frame it was written)
- [x] Applied to all 6 free-floating windows from `roadmap-ux.md` ┬¦3: `WelcomePanel`, `GuidedTour`, `DemoWizardPanel`, `QuickExportPanel`, `HelpPanel`, `CommandPalette` ŌĆö each now calls `Elevation.DrawShadow` immediately after a successful `Begin()`. **Not applied to toasts**: `ToastManager`'s windows use `AlwaysAutoResize` with position computed *before* the size is known for that frame's stack layout, making a same-frame shadow position slightly awkward to get right; descoped as a minor, low-visibility polish gap (toasts are small, transient, and already have a fade animation carrying visual weight)
- [x] `Elevation.PushDockedBorder()`/`PopDockedBorder()` (1px, subtle border color, no shadow ŌĆö flush-docked panels don't "float") applied around all 4 docked windows in `EffectsPanel.cs` (`Navigation`, `Preview`, `Inspector`, `Timeline Dock`) so the dockspace reads as a deliberately bordered layout rather than default-ImGui windows snapped together

## 2. Motion & feedback

### 2.1 Systematic motion, not just toasts
- [x] New `Gui/Motion.cs`: `EaseIn`/`EaseOut`/`Smooth`/`Smoother` ŌĆö the exact same 4 formulas already used by `TimelineEngine.ApplyEffects`'s curve interpolation (`t*t` / `t*(2-t)` / smoothstep / smootherstep), transcribed verbatim rather than reinvented, per this item's own instruction to reuse the existing math. Also a reusable `FadeTracker` class: given any changing "key" object (a section name, a step index, a boolean-as-string), it tracks when the key last changed and returns an eased 0ŌåÆ1 value over a configurable duration ŌĆö a single generic primitive that covers both "content swapped" fades and "just appeared" fades (an appearing window just uses a key that goes from `null` to a constant once, which naturally triggers exactly one fade-in)
- [x] Applied via `ImGuiStyleVar.Alpha` push/pop (confirmed safe: ImGui alpha is purely visual, it does not block interaction, so a fading-in button stays clickable through the transition) to: `EffectsPanel`'s Inspector content (keyed on `_selectedSection` ŌĆö switching Navigation sections now cross-fades instead of cutting), `GuidedTour` (keyed on step index), `DemoWizardPanel` (keyed on step), `WelcomePanel` (keyed on a constant "shown", so it fades in once on first appearance). **Not applied**: the layout-preset switch (`NativeDockBuilder.ApplyPreset`) ŌĆö this rebuilds the native ImGui dock tree via direct `igDockBuilder*` calls (`roadmap-ux.md` ┬¦1.2); there's no supported way to interpolate a dock node's position/size mid-rebuild without fighting the docking system's own layout solver, and attempting it risked exactly the kind of native-interop fragility that area of the code already carries. A hard cut on preset switch (a rare, deliberate action, not a moment-to-moment interaction) is the right trade-off here, not a gap
- [x] Hover/press color-swap evaluated and **declined**, as this item itself anticipated: `Theme.AccentHover`/`AccentActive` instant recolor is already immediate, unambiguous feedback across ~150+ buttons in a dense control panel; a per-frame lerp would add real complexity (state tracking per-widget-id, since ImGui buttons aren't persistent objects) for a difference that's genuinely hard to perceive at typical UI recolor speeds. Closed by verifying there was no real problem worth solving, matching the precedent set by `roadmap-ux.md` ┬¦2.2's FBO-merge audit ŌĆö not implemented, and correctly so

## 3. Component-level craft

### 3.1 Sliders & numeric controls
- [x] `UiUtils.SliderWithReset` keeps `ImGui.SliderFloat` for the actual interaction (click/drag/keyboard-nudge/ctrl-click-to-type/double-click-reset all already work correctly there ŌĆö reimplementing that reliably for ~60 call sites was a real, unjustified risk, not a shortcut) but now draws a thin **delta strip** directly beneath every slider: a track showing the filled region between the current value and the default value, plus a distinct marker line at the default position ŌĆö so "have I strayed from neutral, and by how much" is visible at a glance without pressing "R" to find out. **Technical constraint found and worked around**: true "filled-from-left" track coloring (the Ableton/Figma look the item asked for) isn't achievable by drawing on top of a stock `ImGui.SliderFloat` ŌĆö ImGui draws the frame background and grab handle as one internal call, so anything drawn afterward sits on top of and obscures the grab/text rather than blending underneath it. The delta strip is the honest alternative: real new information (fill + default marker), correctly layered, at low risk ŌĆö not the literal treatment envisioned, but the closest achievable without patching ImGui internals
- [x] Group-header treatment: new `Elevation.BeginCard()`/`EndCard()` (added alongside `Elevation`'s existing shadow/border helpers, since it's the same underlying technique ŌĆö z-order-correct background drawing) using `ImDrawListPtr.ChannelsSplit`/`ChannelsMerge` (confirmed present in this ImGui.NET version via assembly reflection before relying on it) to draw a real bordered background *behind* already-drawn content, not just a color wash on top. Applied to all 9 `EffectsPanel.DrawEffectSliders` groups (Color & Light through Shader-Specific) ŌĆö each now reads as a distinct card, replacing the old `ImGui.Text` + `ImGui.Separator` flat list
