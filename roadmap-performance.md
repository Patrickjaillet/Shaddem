This document goes further and narrower: its only goal is **maximum runtime performance and real system-resource management so shaders stay smooth on weak hardware** ŌĆö integrated-GPU laptops, older machines, hybrid-graphics systems that might silently render on the wrong GPU. `roadmap-ux.md` ┬¦2 proved the *measurement infrastructure* (profiler, benchmark harness, budget constants); this document spends that infrastructure on closing the specific gaps that stand between "fast on this development machine" and "actually protects a genuinely weak machine from stuttering."

> Same living-document rule as every other roadmap here: check boxes and log decisions in the same change that makes the change. Do not let this drift out of sync with the code.

**Baseline audit (2026-07-10, from the actual current code, not assumptions)**:
- **VSync is hardcoded on** (`Rendering/GlWindow.cs`, `options.VSync = true` at window construction) ŌĆö not a setting, not overridable. No frame-rate cap exists anywhere in the codebase if it were ever disabled.
- **No focus/minimize throttling exists at all.** `GlWindow`'s render loop calls the full `RenderFrame`/`ImGuiRender` path every tick regardless of window focus or minimize state ŌĆö Silk.NET's `IWindow.FocusChanged`/`IsVisible` events are never subscribed to. Tabbing away or minimizing burns full GPU/CPU (and laptop battery) for a window nobody is looking at.
- **No GPU capability detection exists.** Nothing in the codebase ever calls `GL_RENDERER`/`GL_VENDOR`/`GL_VERSION`, and there is no "low-end"/"integrated GPU" heuristic or automatic quality-tier selection anywhere.
- **`AdaptiveResolutionController` (`roadmap-ux.md` ┬¦2.2) defaults to `Enabled = false`** and is only reachable via a manual checkbox buried in `PerformanceHud`. On a genuinely weak GPU, the first too-heavy shader stutters at fixed resolution until a user happens to find and enable that checkbox themselves.
- **96 of the ~200 bundled shaders hardcode raymarching iteration/step counts as GLSL `#define` compile-time constants** (e.g. `shaders/Mandelbulb_I.glsl`: `MAX_STEPS 160`, `ITERATIONS 7`) ŌĆö every shader always runs at full complexity regardless of detected hardware; the most GPU-expensive shaders in the library are exactly the ones with zero fallback for weak machines.
- **Two real synchronous-main-thread stalls exist beyond what `roadmap-ux.md` ┬¦2.3 already found and fixed**: `AudioAnalyzer.AnalyzeFull`'s full-track offline FFT runs synchronously on the UI/GL thread from both the drag-drop handler and `AudioPanel`'s Load & Play button (blocks the entire render/input loop for the whole analysis duration on a multi-minute track), and `VideoRecorder.WriteFrame`'s synchronous ffmpeg-stdin write (already flagged, not fixed, in `roadmap-ux.md` ┬¦2.3) remains open.
- **`TimelineEngine.GetFilteredActiveClips` is re-evaluated 6 separate times per frame** across the different `Apply*` methods (`ApplyEffects`, `ApplyShader`, `ApplyImageLayers`, `ApplyModelLayers`, `ApplyMusicClip`, `ApplyLayerAutomation`), each a fresh LINQ/`yield return` pass over the full clip list, and `ApplyEffects` does live reflection (`typeof(EffectParams).GetField(...)`) plus a fresh string `Split`+`Select`+`Concat` for every active effect clip, every single frame ŌĆö real, measurable per-frame GC pressure that has never been benchmarked.
- **No multi-GPU/hybrid-graphics awareness exists.** Grepped the entire repo for the standard `NvOptimusEnablement`/`AmdPowerXpressRequestHighPerformance` exported-symbol convention that tells Windows to prefer a laptop's discrete GPU over its integrated one ŌĆö zero matches. On hybrid-graphics laptops (a large share of the "petites configurations" this roadmap targets), Windows may default this process to the weaker integrated GPU, silently undermining every other item below before it gets a chance to matter.
- **Window resolution defaults are hardcoded** (`Settings/AppSettings.cs`: `WindowWidth = 1920`, `WindowHeight = 1080`), not derived from the detected display's actual resolution or refresh rate.

None of this is duplicate scope with `roadmap-ux.md` ┬¦2 ŌĆö that section built the measurement tools and shipped the first, deliberately opt-in/conservative pass; this document uses those same tools to go further and make the smooth-on-weak-hardware behavior automatic rather than something a user has to discover.

## 0. Guiding principles (non-negotiable)

- [ ] Every optimization here is validated with a real, numeric before/after measurement using the existing `GpuProfiler`/`PerformanceHud`/`--benchmark` infrastructure ŌĆö never assumed, never shipped on "this should be faster" reasoning alone.
- [ ] Prefer automatic detection over manual opt-in wherever it can be done safely ŌĆö a user on a weak laptop should get smooth playback by default, not have to discover a checkbox in a HUD they may never open.
- [ ] No optimization silently reduces visual quality on capable hardware. Quality scaling only activates when a real measurement says it's needed (detected low tier, or live frame-time budget overrun), and is always visible/overridable, never a silent, permanent downgrade.
- [ ] Honesty about what can and can't be verified in this environment: this development machine is not a genuinely weak machine. Every item that claims a low-end-hardware benefit must say plainly whether it was measured here and reasoned to generalize, or whether it genuinely cannot be verified without real weak hardware ŌĆö flagged, not assumed, matching the discipline already established in `roadmap-features.md`'s decision log.
- [ ] All UI text stays English-only; no comments in source beyond the one copyright header line ŌĆö same project-wide rules as every other roadmap.
- [ ] Re-run the full existing regression suite (every `--test-*`/`--benchmark` harness already in this project) after every phase ŌĆö no exceptions, no silent skips.

## 1. Hardware detection & automatic quality tiers

- [x] Query `GL_RENDERER`/`GL_VENDOR`/`GL_VERSION` once at startup and log it via `AppLog.Info` ŌĆö `GlWindow.OnLoad` now calls `Api.GetStringS(StringName.Renderer/Vendor/Version)` right after `Api = _window.CreateOpenGL()`, the earliest point a valid context exists, exposed as new public `GlWindow.GpuRenderer/GpuVendor/GpuVersion` properties. Verified on the real machine: `GPU: NVIDIA GeForce GT 1030/PCIe/SSE2 (NVIDIA Corporation), GL 3.3.0 NVIDIA 582.66` appears in `logs/shaderdemo.log` on every launch.
- [x] Defined `QualityTier` (`Unknown/Low/Medium/High`, new `Rendering/QualityTier.cs`) and a `QualityTierDetector` (new `Rendering/QualityTierDetector.cs`) combining a renderer-string heuristic (Intel HD/UHD/Iris, "Microsoft Basic Render Driver") with a real micro-benchmark run once during the existing splash-screen loading phase in `Program.cs` (renders `shaders/Mandelbulb_I.glsl` ŌĆö the roadmap's own example of a heavy raymarcher ŌĆö for 25 warmup + 20 sampled frames before the real GUI appears). **Real finding that changed the design**: the existing `GpuProfiler`'s `GL_TIME_ELAPSED` timer queries, while proven working earlier this session (`roadmap-features.md`'s `LayerBenchmarkTest`), turned out to **never return an available result on this machine when queried this way** ŌĆö verified by instrumenting `GpuProfiler.Section.End()` directly and observing `available=0` for 30+ consecutive frames across every profiled section, not just the benchmark one; root cause not conclusively identified (driver/timing-specific, not a logic bug ŌĆö the exact same code path worked in an earlier session on the same GPU), and not worth chasing further given a low-end-hardware detection feature needs to work even when GPU timer queries are unreliable, which is itself a realistic scenario on weaker/older hardware. **Designed a real fallback instead of silently defaulting**: `QualityTierDetector.ClassifyFromGpuBenchmark` (preferred, used when the timer query does produce samples) sits alongside a new `ClassifyFromCpuFallback`, which uses the real per-frame `deltaSeconds` Silk.NET already passes into `window.RenderFrame` (genuine VSync-paced wall-clock frame time, not raw GL-submission time, which earlier `LayerBenchmarkTest` work in this session already found to be near-meaningless under VSync) as the fallback signal ŌĆö conservatively only escalates to `Low` when frame time clearly exceeds budget, defaults to `Medium` otherwise since CPU wall-clock alone can't reliably distinguish Medium from High once VSync caps it. Verified end-to-end on the real machine: correctly detected **Low** tier via the CPU fallback path (`111.01 ms avg over 20 samples`, ~9x over budget ŌĆö genuinely slow for this GPU on a 160-step raymarcher), with the log line honestly stating `CPU wall-clock fallback (GPU timer query unavailable on this driver/GPU)` rather than silently presenting it as a GPU measurement.
- [x] Persisted the detected (or user-overridden) tier in `AppSettings` (`QualityTier`, `DetectedGpuName`, both new fields ŌĆö auto-persisted by the existing `SettingsService` JSON round-trip, no changes needed there). Detection is skipped entirely on subsequent launches once `QualityTier != Unknown` (verified: a second real launch shows the `GPU:` log line but no `Quality tier detected` line ŌĆö confirmed via timing that the fast path genuinely skips the benchmark). Changeable anytime in `SystemPanel` (Settings): a "Quality Tier" combo box for direct override (saves immediately) plus a "Re-detect on Next Launch" button that resets to `Unknown` so detection re-runs next time ŌĆö both wired and building cleanly.
- [x] Surfaced "Detected: `<tier>` (`<GPU name>`)" at the top of `PerformanceHud` (`PerformanceHud.Draw` signature gained an `AppSettings` parameter; the one real call site in `Program.cs` and the one test-harness call site in `EffectsPanelTest.cs` both updated) ŌĆö visible and debuggable, not silent.

## 2. Make adaptive resolution actually protect low-end hardware

- [x] `AdaptiveResolutionController` now auto-enables on Low tier and stays off on Medium/High, preserving prior behavior everywhere else. New `QualityTierDetector.ApplyTierDefaults(manager, tier)` sets `AdaptiveResolution.Enabled`/`MinScale` in one place, called from three sites so it's never stale: `window.Load` (fast path ŌĆö tier already known from a previous run), `FinishTierDetection` (first-run ŌĆö right after a tier is actually determined), and `SystemPanel`'s Quality Tier combo handler (so a manual override takes effect immediately, not just on next launch). Still manually overridable afterward via the existing `PerformanceHud` checkbox, since `ApplyTierDefaults` only sets the *initial* state, nothing fights the user's own toggle for the rest of the session. Verified with a new `--test-tier-defaults` harness asserting all three tiers produce the exact expected `Enabled`/`MinScale` combination against a real `ShaderManager`.
- [x] Lowered the Low-tier render-scale floor from the shared 0.5├Ś to a dedicated `AdaptiveResolutionController.LowTierMinScale = 0.35├Ś` (0.5├Ś kept as `DefaultMinScale` for Medium/High, unchanged). `MinScale` was promoted from a `private const` to a settable property so `ApplyTierDefaults` can select between them per tier without forking the whole class.
- [x] Re-evaluated `roadmap-ux.md` ┬¦2.2's full-resolution bloom/accumulation buffers with a real before/after measurement, per the item's own explicit instruction ŌĆö and the measurement said **don't make this change**. New `--test-posteffects-cost [renderScale]` harness renders `Mandelbulb_I.glsl` (the same reference shader as ┬¦1) and toggles Bloom/`FeedbackOpacity` on and off, using the same VSync-paced `deltaSeconds` timing ┬¦1 already established as reliable on this machine (GPU timer queries remain unreliable here, so this reused the proven fallback rather than the unreliable primary path). Measured at the actual new Low-tier floor (`RenderScale=0.35`, internal `672├Ś378` against a `1920├Ś1080` output): `Bloom=0,Feedback=0` ŌåÆ 19.9 ms avg, `Bloom=0.6,Feedback=0.3` ŌåÆ 19.8 ms avg ŌĆö a ~0.1 ms spread, well inside measurement noise. Re-measured at full resolution (`RenderScale=1.0`, `1920├Ś1080`) as a sanity check on the methodology itself: 110.9 ms vs 112.5 ms ŌĆö same ~1.5 ms/1.3% spread, confirming the earlier result wasn't a floor-specific fluke. **Conclusion, not assumption**: for this GPU and this reference shader, raymarching cost so thoroughly dominates total frame time (ALU-bound, independent of the small fixed-size bloom/accumulation buffers, which are bandwidth-bound and comparatively cheap) that scaling those buffers down would not move the measured number in any way that matters ŌĆö implementing the change would add real complexity (a new resize hook on `PostEffectsPipeline` and `ShaderManager`'s accumulation buffers, per the research that scoped this item) for no measured benefit. **Left deliberately unimplemented, not silently skipped**: this is the correct application of the project's own "measurement first" principle ŌĆö sometimes the measurement's answer is "don't."

## 3. Frame pacing, VSync, and power-saving

- [x] `GlWindow` now subscribes to Silk.NET's `IView.FocusChanged`/`IWindow.StateChanged` events (confirmed via reflection over the actual interface hierarchy ŌĆö `FocusChanged` lives on `IView`, not directly on `IWindow`, since the earlier baseline-audit wording undersold exactly where it was) and drops both `FramesPerSecond`/`UpdatesPerSecond` to a throttled `5.0` when the window is unfocused or minimized (`WindowState.Minimized`), restoring the normal rate immediately on focus/visibility regain via a new `ApplyFrameRateState()` helper and public `IsThrottled` property (also surfaced as a warning line in `SystemPanel` so it's visible, not silent). **Honest verification limit**: focus/minimize are OS window-manager events that a headless CLI harness cannot synthesize ŌĆö the new `--test-frame-pacing` harness verifies the surrounding contract (initial state, `SetVSync` round-trip) but the throttle-on-focus-loss path itself is verified by code review against the same event-subscription pattern already proven for `Resize`/`Closing` in this file, not by triggering a real OS focus change in an automated test.
- [x] VSync is now a real setting: new `AppSettings.VSync` (default `true`, preserving today's exact behavior), threaded into `new GlWindow(..., vsync: appSettings.VSync)`, with a live `GlWindow.SetVSync(bool)` method (not just a constructor-time flag ŌĆö Silk.NET's `IViewProperties.VSync` is genuinely runtime-settable, confirmed via reflection) exposed as a checkbox in `SystemPanel`. Whenever VSync is off, a new `QueryPrimaryRefreshRate()` (mirrors the existing `QueryContentScale` GLFW-query pattern already in this file) sets `FramesPerSecond`/`UpdatesPerSecond` to the primary display's actual refresh rate as a fallback cap ŌĆö so turning VSync off for input-latency reasons can never accidentally uncap to hundreds/thousands of fps. Verified end-to-end with `--test-frame-pacing`: `SetVSync(false)` ŌåÆ `SetVSync(true)` round-trip confirmed correct on a real window, both directions.

## 4. Shader complexity scaling (LOD) for the most expensive shaders

- [x] Extended `--benchmark` (`BenchmarkTest.cs`) to bucket frame times per shader name alongside the existing aggregate stats, printing a "top 15 slowest by avg frame time" breakdown at the end. **Real methodology finding while using it**: a full 201-shader, 205-second run revealed the per-shader numbers were dominated by a one-time ~100msŌĆō2000ms stall on each shader's *first* rendered frame after switching (consistent with driver-level shader-backend JIT deferred to first draw, not link time ŌĆö the same class of cold-start cost `roadmap-ux.md` ┬¦2.2's shader-binary-cache investigation found at the process level, recurring here per-shader) ŌĆö with only 1ŌĆō3 samples collected per shader in the existing 1-second cycling window, the numbers reflected switch-stall cost, not steady-state rendering cost, and were the wrong signal for LOD purposes. Documented as a genuine discovery, not silently worked around: `--benchmark`'s per-shader breakdown is still useful (and now shipped) for spotting switch-stall-heavy shaders, but a **second, dedicated harness** (`--test-shader-lod-survey`, new `ShaderLodSurveyTest.cs`) was built to get the steady-state signal ┬¦4 actually needs: 15 warmup frames discarded (absorbing the JIT stall) + 15 sampled frames averaged, per shader, across the full 201-shader library, using the real VSync-paced `deltaSeconds` timing already proven reliable in ┬¦1ŌĆō┬¦3 (not GPU timer queries, still unreliable on this machine, and not a raw `Stopwatch` around `RenderFrame()`, already known from earlier session work to be near-meaningless under async GL submission). Real result at 1920├Ś1080 on the available GPU: slowest shaders ranged 177ŌĆō672 ms/frame (1000%ŌĆō4000% over the 16.7 ms budget), a genuine, measured outlier list ŌĆö not a guess from filenames.
- [x] Using that real data, added a new `uniform float customQualityScale` (`ShaderWrapper.cs`, alongside the existing `customStrobe`/`customRgbSplit`/etc. custom-uniform mechanism ŌĆö reusing the established injection pattern, not inventing a parallel one) driven by `AdaptiveResolutionController.CurrentScale` (new optional `qualityScale` parameter on `RenderPipeline.RenderShaderPass`, wired from all 5 call sites in `ShaderManager`) ŌĆö deliberately tied to the *live* adaptive-resolution signal from ┬¦2 rather than a static tier lookup, so shader complexity only drops when the system is actually under measured load (defaults to `1.0`/today's exact behavior whenever adaptive resolution isn't active, which is always true on Medium/High and true on Low tier except when genuinely struggling). Converted the **5 most extreme outliers** from the survey (not all ~96 raymarch-named shaders ŌĆö sized honestly from real data, per this item's own instruction): `Electric_Tunnel.glsl` (`VOLUMETRIC_STEPS`/`SAMPLES`), `Filaments_Electriques.glsl` (packed-vector step count), `Space_I.glsl` (`STEPS`), `Spider_Mouse.glsl` (`WEB_DENSITY`, an O(n┬▓) double-loop density knob), `Falling_cubes_of_ice.glsl` (two separate unnamed loop literals ŌĆö primary raymarch cap and a distinct shadow-ray cap). Each required reading the actual shader to find its real complexity knob, since naming/structure varied per file (`#define` constants, packed `vec4` fields, and raw literals all appeared) ŌĆö confirming this genuinely needed per-file judgment, not a mechanical find-replace. All 5 verified to still compile and select correctly via `--test-shader`. **Measured real before/after** with a new `--test-shader-quality-scale <name>` harness (renders at `CurrentScale=1.0`, then forces `AdaptiveResolutionController` down via repeated over-budget `Update()` calls to `0.5`, sampling steady-state `deltaSeconds` both times): `Electric_Tunnel` ŌłÆ74.6%, `Space_I` ŌłÆ68.2%, `Spider_Mouse` ŌłÆ44.5%, `Filaments_Electriques` ŌłÆ19.0%, `Falling_cubes_of_ice` ŌłÆ2.9%. **Honest outlier, not hidden**: `Falling_cubes_of_ice`'s near-zero improvement was investigated, not shrugged off ŌĆö its primary raymarch loop exits early via a distance-based `break` well before reaching its step cap in practice (so scaling the cap did nothing), and after reading the full shader, its actual dominant cost turned out to be a *separate*, initially-unscaled 32-step shadow-ray loop plus an unconditional 4-sample normal-calculation block; the shadow loop was found and also converted (small further edit, verified with a second before/after run), improving the result from 0% to a real but modest 2.9% ŌĆö the remaining cost lives in the unscaled normal-calculation block, which would need a more invasive rewrite than this pass's scope to also fix, and is left explicitly as future work rather than silently left broken or fixed by inflating the claim. The remaining ~90+ raymarch-named shaders in the library, including 10+ more from the survey's own top-20, are **not** converted ŌĆö real scope for a future pass, sized from data collected in this one.

## 5. Offload remaining synchronous CPU work off the render thread

- [x] Moved `AudioAnalyzer.AnalyzeFull`'s full-track offline FFT to a background `Task` via a new `AsyncAudioAnalyzer` (new `Core/Audio/AsyncAudioAnalyzer.cs`), mirroring the already-proven `AsyncTextureLoader` pending-set + `ConcurrentQueue` + main-thread-drain pattern rather than inventing a new one. Wired into `ShaderManager` (new `AsyncAudio` property) and all three real call sites (the drag-drop handler and both "Load & Play" buttons ŌĆö `AudioPanel.cs` and, discovered during implementation, a third site in `DemoWizardPanel.cs`'s wizard flow that the original item didn't know about), each now firing `RequestAnalysis` plus an "analyzing..." `ToastManager` toast instead of blocking. `Program.cs`'s `UpdateFrame` handler drains completed results every frame and applies `SetAnalysis` (a GL/main-thread-only call) plus a completion toast, keeping all GL-touching work off the background thread.
- [x] Pooled `LiveAudioAnalyzer.AnalyzeWindow`'s `Complex[1024]` FFT buffer as a persistent `_fftBuffer` field instead of allocating fresh every ~20ms callback ŌĆö safe because the method already fully resets both real (`.X`) and imaginary (`.Y`) parts every call and is already called under `lock (_lock)` from `OnDataAvailable`, guaranteeing single-threaded reuse.
- [x] Investigated `ParticleSystem`'s per-frame buffer upload with a real before/after measurement using the existing `--test-particle-benchmark` harness ŌĆö and the measurement said the opposite of what conventional GL wisdom predicted. Implemented `BufferSubData` (renamed the old single `UploadBuffer` into `AllocateBuffer`/`BufferData`, called only from `Resize`, plus a new `UpdateBuffer`/`BufferSubData` for the per-frame `Update` path), benchmarked it, then ran a controlled A/B/A comparison (BufferSubData ŌåÆ BufferData ŌåÆ BufferSubData again, same session, to rule out ordering/noise) to be sure: `BufferSubData` was **consistently slower** on this GPU/driver (e.g. 50000 particles: 5.18ŌĆō5.38 ms vs. 2.42ŌĆō2.79 ms with `BufferData`, roughly 2x), reproducible across three separate runs. **Reverted rather than shipped**: `Update()` now calls the same `BufferData`-based upload as `Resize()` (collapsed back to a single `UploadBuffer` method; the dead `BufferSubData` path was deleted rather than left unused, per this project's no-dead-code convention). Final confirming run after the revert: 50000 particles = 1.53ŌĆō2.42 ms across two clean runs, consistent with the original `BufferData` baseline and clearly faster than the abandoned `BufferSubData` path ŌĆö a genuine "tried, measured, reverted" finding, not a silently dropped item.
- [x] Closed `roadmap-ux.md` ┬¦2.3's long-flagged synchronous ffmpeg-stdin write. `VideoRecorder` now runs a dedicated background writer thread (`RunWriterLoop`) reading from a `BlockingCollection<byte[]>` (bounded to 8 queued frames ŌĆö `WriteFrame` enqueues instead of writing/flushing inline on the GL thread that calls it from `ShaderManager.RenderFrame`). `Stop()` calls `CompleteAdding()` and `Join()`s the writer thread before closing `StandardInput`'s stream, so no frame is dropped or the stream closed mid-write. A broken pipe (ffmpeg process gone) is detected on the writer thread, flagged via a `volatile _pipeBroken` field, and actually torn down (mirroring the old inline behavior of auto-stopping recording) on the *next* `WriteFrame` call from the GL thread ŌĆö deliberately not calling `Stop()` from the writer thread itself, since `Stop()` joins that same thread and would deadlock. **Honest verification limit**: this environment still has no ffmpeg installed (re-confirmed via `where ffmpeg`, same finding as `roadmap-features.md` ┬¦1.6), so the actual recording path cannot be exercised end-to-end here ŌĆö verified by code review and a clean full-solution build instead, following the same "flagged, not assumed" discipline as ┬¦7's multi-GPU item.

## 6. Reduce per-frame GC pressure in the timeline/effects update path

- [x] Consolidated `TimelineEngine`'s six separate per-frame `GetFilteredActiveClips` calls into one. `GetFilteredActiveClips` changed from a lazy `yield return` `IEnumerable<TimelineClip>` (recomputed on every enumeration) to an eagerly-built `List<TimelineClip>`, and each of the six `Apply*` methods was split into its original public single-argument overload (unchanged signature, still computes its own list ŌĆö kept for the existing individual test-harness call sites: `TimelineImageLayerTest`, `TimelineGapsTest`, `MegademoShowcaseTest`, `FullFeatureTest`) plus a new private overload taking a pre-computed `List<TimelineClip> activeClips`. A new public `ApplyAll(ShaderManager manager, EffectParams effects, double currentTime)` computes the active-clip list exactly once and calls all six private overloads with it. `Program.cs`'s real per-frame hot path (the only place all six were ever called together every frame) now calls `timeline.ApplyAll(...)` instead of six separate calls.
- [x] Cached `ApplyEffects`'s reflection lookup via a new `private static readonly Dictionary<string, FieldInfo?> _effectFieldCache` and a `GetEffectField(string resource)` helper ŌĆö the `ToPascalCase` `Split`+`Select`+`Concat` and `typeof(EffectParams).GetField(...)` reflection call now only run once per distinct clip-resource string ever encountered for the lifetime of the process, not once per active effect clip per frame. Not thread-safe by design, matching the project's existing single-threaded-update-loop assumption used elsewhere (e.g. `LiveAudioAnalyzer`'s pooled FFT buffer in ┬¦5).
- [x] Measured the real before/after with a new `--test-gc-pressure` harness (`GC.CollectionCount(0/1/2)` deltas over 300 sampled frames, after a 50-frame warmup, with 20 active effect clips ŌĆö one per `EffectParams` field ŌĆö plus 1 active shader clip, the worst realistic case for both fixes at once). Getting an honest "before" number required actually reverting both fixes temporarily (not just reasoning about it): reflection caching was hand-reverted to the original inline `GetField`/`ToPascalCase` call and a throwaway harness called the six original individual `Apply*` methods (proven to still compile against pre-fix `TimelineEngine.cs` since their public signatures never changed), isolating the combined effect of both changes rather than assuming it. **Measured result**: `Gen0=2, Gen1=0, Gen2=0` over 300 frames before, `Gen0=0, Gen1=0, Gen2=0` after ŌĆö a small but real and fully eliminated Gen0 allocation count in this specific stress scenario (heavier real-world timelines with more simultaneously active clips, or lower-end hardware running the same content for longer, would scale this further; the item's own honesty principle from ┬¦0 applies: this was measured on the available development machine's timeline-heavy synthetic case, not on genuinely weak hardware).

## 9. Suggested phasing
